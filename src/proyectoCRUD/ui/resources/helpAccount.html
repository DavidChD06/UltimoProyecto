Bank Account Management Application

This application allows for the management of bank accounts associated with a customer, including the creation, editing, deletion, and viewing of accounts, as well as navigation to the movements of a specific account.

1. General Structure

The application is developed in JavaFX following a client-server pattern, where the graphical interface communicates with a REST backend via the AccountRESTClient class.

The main view displays a table (TableView) with the accounts associated with the authenticated customer.

2. Account Table

The table displays the following information for each account:

  • ID: Unique identifier of the account (read-only).
  • Description: Account description (editable).
  • Type: Account type (STANDARD or CREDIT).
  • Begin Balance: Initial balance.
  • Balance: Current balance (read-only).
  • Credit Line: Credit line (editable only for credit-type accounts).
  • Begin Balance Timestamp: Creation date.

3. Data Editing

3.1 Description

The description can be edited directly from the table. The system validates that it is not empty and, in the case of an existing account, it is automatically updated on the server.

3.2 Begin Balance and Balance

When modifying the Begin Balance, the Balance field updates immediately with the same value. This ensures consistency between both fields from the start.

The balance is not manually editable by the user.

3.3 Credit Line

The Credit Line field is only editable when the account type is CREDIT. For STANDARD accounts, this field remains locked.

The application validates that the entered value is not negative before accepting it.

4. Creating New Accounts

Account creation is done via a toggle button (Add):

  • When activated, a new account is created with default values.
  • The table enters edit mode.
  • The user must complete the required fields.
  • Upon deactivating the button, the account is validated and sent to the server.

During this process, the new account is kept in memory and is not updated on the server until its creation is confirmed.

5. Update and Persistence

For existing accounts, any modification made from the table is automatically sent to the backend via REST calls.

This ensures that changes are reflected immediately in the database.

6. Data Refresh

The Refresh button requests all accounts associated with the customer from the server again and reloads the table with the updated data.

This mechanism ensures that the view is synchronized with the actual server state.

7. Deleting Accounts

An account can only be deleted if it has no associated movements. Before proceeding, the application requests confirmation from the user.

8. Navigation to Movements

Upon selecting an account and pressing the Movements button, the application opens a new window displaying the movements corresponding to the selected account.

The account is explicitly passed to the movements controller, ensuring that incorrect data is not worked with.

9. Conclusion

The application provides complete and secure bank account management, combining interface validations, automatic data updates, and efficient communication with the REST backend.
